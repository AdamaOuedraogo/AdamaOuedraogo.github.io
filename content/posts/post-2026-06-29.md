## 1. Short post

Your test suite already knows how to run tests, parse reports, and query tickets. The question is: can your AI agent reach those capabilities?

That's what MCP unlocks. It lets you expose your own QA stack as callable tools an agent can discover and use — not generic AI, but *your* testing logic.

Here's a reflection worth sitting with: if you had to expose 3 of your real QA tasks as MCP tools, which 3 would deliver the most value to your team?

For me, the obvious candidates are: run a test suite, parse a failure report, and query open tickets. Those three alone change what an agent can actually do for quality.

Which three would you pick?

## 2. Storytelling post

I expected building an MCP server to be hard. It wasn't — and that surprised me more than any tutorial could.

I sat down to code a tiny server with a single QA-relevant tool: something like `check_url_status(url)` returning an HTTP status code. My assumption was that I'd spend the hour wrestling with the protocol — connection handling, negotiation, JSON-RPC plumbing.

Instead, the SDK handled almost all of that. My job was just to write the logic of the tool. The framework took care of the protocol so I could focus on what actually mattered: what the tool *does*.

That reframed the whole exercise for me. The hard part of agentic QA isn't the wiring. It's deciding which of your testing capabilities deserve to become tools an agent can call — running tests, parsing reports, querying tickets.

If the plumbing is mostly solved, the real skill becomes design: what do you expose, and why? What's the first QA capability you'd hand to an agent?

## 3. Educational post

Most QA engineers think AI agents need access to "the AI." They actually need access to *your* tools.

That's the core idea behind MCP (Model Context Protocol). It lets you take a function — say, one that counts failed tests in a report or checks an HTTP status — and expose it as a tool an LLM can discover and call.

The anatomy is simpler than it sounds. With a framework like FastMCP, you write the function's logic, mark it as a tool, and the SDK handles the protocol: connection, negotiation, JSON-RPC. You concentrate on the testing logic, not the wiring.

Why this matters for QA: every agentic workflow you might build or sell depends on this one foundational skill — exposing your testing capabilities as tools. Run tests. Parse reports. Query tickets. An agent is only as useful as the tools you give it.

So before chasing fancy agent demos, start here: pick one real QA task you do often, and imagine it as a callable tool. Which task would you expose first?
